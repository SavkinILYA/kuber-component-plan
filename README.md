## Расчёт требуемых ресурсов

### Шаг 1 — Ресурсы приложения

| Компонент | RAM на экземпляр | CPU на экземпляр | Копий | Итого RAM | Итого CPU |
|---|---|---|---|---|---|
| База данных | 4 GB | 1 | 3 | 12 GB | 3 |
| Кеш | 4 GB | 1 | 3 | 12 GB | 3 |
| Фронтенд | 50 MB | 0.2 | 5 | 250 MB | 1 |
| Бекенд | 600 MB | 1 | 10 | 6 GB | 10 |
| **Итого** | | | | **~30.25 GB** | **17** |

### Шаг 2 — Запас на overhead Kubernetes

Kubernetes резервирует ресурсы под системные компоненты (kubelet, kube-proxy, CoreDNS, CNI плагин и др.). Добавляем **+30%**:

- RAM: 30.25 × 1.3 ≈ **40 GB**
- CPU: 17 × 1.3 ≈ **22 CPU**

### Шаг 3 — Запас на отказ одной ноды

При выходе из строя одной worker ноды её нагрузка перераспределяется на оставшиеся. Кластер должен справляться при N-1 нодах.

**Выбор конфигурации worker нод: 4 ноды по 12 GB RAM / 8 CPU**

| Сценарий | Доступно RAM | Доступно CPU | Достаточно? |
|---|---|---|---|
| Все ноды работают (4×12) | 48 GB | 32 | да |
| Одна нода упала (3×12) | 36 GB | 24 | да (>40 GB с запасом не нужен при N-1) |

При отказе одной ноды 36 GB > 30.25 GB (реальная нагрузка) — кластер справляется.

### Шаг 4 — Служебные ресурсы для master нод

Master ноды запускают: kube-apiserver, kube-scheduler, kube-controller-manager, etcd. Для HA нужно нечётное количество master нод (кворум etcd).

**3 master ноды по 4 GB RAM / 2 CPU** — стандартная конфигурация для production.

## Итоговая конфигурация кластера

| Тип ноды | Количество | RAM | CPU | Назначение |
|---|---|---|---|---|
| Master | 3 | 4 GB | 2 | Control plane, etcd (HA, кворум) |
| Worker | 4 | 12 GB | 8 | Рабочая нагрузка приложения |
| **Итого** | **7** | **60 GB** | **38** | |

### Обоснование

- **3 master** — нечётное количество обеспечивает кворум etcd (2 из 3 нод должны быть доступны)
- **4 worker** — при выходе одной ноды оставшиеся 3 × 12 GB = 36 GB покрывают реальную нагрузку 30.25 GB
- **12 GB / 8 CPU на worker** — даёт достаточно места для самых тяжёлых компонентов (БД и кеш по 4 GB) с учётом системного overhead
- На каждой worker ноде ~2 GB RAM резервируется под системные компоненты Kubernetes

### Размещение компонентов

База данных и кеш (отказоустойчивые, 3 реплики) рекомендуется размещать по одной реплике на отдельную ноду через `podAntiAffinity` — это обеспечит отказоустойчивость при падении ноды.
